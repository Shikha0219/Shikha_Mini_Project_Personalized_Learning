1. Executive Summary 

This document presents a comprehensive Design Thinking methodology applied to the Automated Data Sensitivity Classification Engine project. The initiative addresses a critical organizational need to classify large volumes of unstructured and semi-structured data by sensitivity level, ensuring regulatory compliance and enabling effective data governance across payment processing clients including Discover, Visa, and American Express. 

Business Critical Challenge: Organizations lack scalable mechanisms to automatically classify sensitive content, resulting in accidental exposure of Restricted/Confidential data, regulatory fines (GDPR up to 4% of global revenue), and vendor prerequisite failures with payments clients. 

Expected Outcome: A production-grade ensemble classification system achieving ≥95% precision on Restricted data with explainability, supporting on-premises deployment, and enabling downstream DLP/governance policy enforcement.

4. Phase 2: Problem Definition & Gap Analysis 

4.1 Current State Analysis 

Operational Reality: 

Manual Classification: Data classification conducted entirely by human reviewers across multiple departments 

Scale Limitations: Cannot process high volumes of unstructured data (customer support tickets, financial records, HR files) 

Consistency Issues: Different reviewers apply inconsistent judgment; no standardized taxonomy 

Compliance Failures: Accidental exposure of Restricted/Confidential data; audit failures 

No Enforcement Mechanism: Data may be classified, but nothing enforces the rules, so sensitive data is still exposed. 

 

4.2 Desired Future State 

Vision: 

Automated Classification: Documents automatically classified into Public | Internal | Confidential | Restricted 

Scalability: Process any volume of unstructured/semi-structured text; batch and real-time streaming 

Explainability: Each prediction includes ≥3 evidence tokens showing which phrases drove classification 

Auditability: Timestamped classification logs; dual-approval labeling; human-in-loop for edge cases 

Policy Enforcement: The system tells security tools what data is sensitive so they can take action. 

Regulatory Compliance: Satisfy PCI DSS, GDPR, CCPA vendor prerequisites for payments clients

5. Phase 3: Ideation & Solution Development 

Design thinking ideation encourages divergent thinking to explore multiple solution pathways, then convergent thinking to evaluate and select the most viable. 

5.1 Solution Overview: Ensemble Classification Architecture 

The proposed solution employs an ensemble machine learning approach combining rule-based pattern matching with deep learning transformers, ensuring both precision and explainability. 

Core Components 

Ingestion Layer: Kafka/S3/REST API connectors. Apache Tika, pdfplumber, python-docx for multi-format parsing. 

Pre-Processing: Text cleaning, PII masking (Presidio library), tokenization, language detection with language-specific routing. 

Feature Engineering: TF-IDF+n-gram baselines. Contextual embeddings via fine-tuned BERT/RoBERTa/DeBERTa on domain corpus. 

Classification Models: Ensemble: Logistic Regression (baseline), XGBoost, Fine-tuned transformer. Voting/stacking layer for final label. 

Explainability Layer: SHAP for tree models, LIME/Integrated Gradients for transformers. Evidence tokens surfaced per prediction. 

Policy Engine: Rule overlay: if Restricted token detected override to Restricted regardless of model score. Configurable threshold table. 

Output/API: REST API (FastAPI). Response: label + score + evidence + action. Webhook for downstream DLP/CASB/data catalog. 

MLOps: MLflow for experiment tracking. Model registry with champion/challenger. Drift monitoring via Evidently AI. Monthly retraining trigger.

8. Industry Context & Vertical Insights 

8.1 Payments Industry Vertical 

Our key clients—Discover, Visa, American Express—operate under strict PCI DSS (Payment Card Industry Data Security Standard), GDPR, and CCPA regulations. These frameworks mandate that vendors demonstrate documented data classification and protection controls. 

PCI DSS Compliance Requirements 

Requirement 3.1: Render PII unreadable anywhere it is stored—masking, encryption mandatory 

Requirement 3.4: Render all card data unreadable during storage 

Requirement 8.2: Use strong authentication; maintain audit trails for data access 

Requirement 10: Implement logging and monitoring; maintain audit logs of access to cardholder data 

GDPR & CCPA Implications 

Data Inventory: Organizations must maintain records of processing activities. Automated classification enables data inventory at scale. 

Data Minimization: Classification informs which data can be deleted or depersonalized, reducing regulatory surface. 

Right to Access: When individuals request their data, classification identifies which records require response. 

Fines & Liability: GDPR fines up to 4% of global revenue for data breaches. Documented classification demonstrates due diligence. 

8.2 Competitive Landscape 

Existing Solutions: Competitors (Microsoft Purview, Symantec DLP, Google Cloud DLP) offer classification, but often as black-box models or require third-party annotations. 

Our Differentiation: Transparency via explainability, on-premise deployment, domain-specific training on payment/financial data, compliance-focused design. 

Market Opportunity: $3.2B+ data governance market; 87% of organizations report data classification as critical need (Forrester 2024). 

 

9. Financial Services Payment Architecture 

9.1 Integration with Payment Processing Pipeline 

The classification engine integrates into payment and financial processing workflows at critical junctures, ensuring sensitive transaction data is protected throughout its lifecycle. 

Ingestion Points 

Customer Support Tickets: Classify immediately on ticket creation; flag Restricted for redaction before agent view 

Financial Documents: Classify invoices, receipts, bank statements on upload; enforce encryption if Restricted 

Merchant Agreements: Classify contracts; identify Restricted payment terms; audit access 

Payment Risk Assessments: Classify risk evaluation reports; ensure only authorized personnel access 

Enforcement Actions 

Public: Allow sharing in marketing, public docs 

Internal: Share within organization; log access 

Confidential: Encrypt at rest/transit; restrict to named roles; MFA required 

Restricted: Quarantine; no export; dual-approval for access; continuous audit; mTLS required for payments clients 

9.2 mTLS Integration for Payments Clients 

Payment processing requires mutual TLS authentication. Our classification API supports mTLS natively, enabling secure credential-based access for Discover, Visa, and Amex integration. 

Certificate Management: Client certificates issued per vendor; rotated annually; revocation managed via CRL 

Authentication: API validates client cert before accepting classification request 

Audit Trail: All mTLS connections logged with certificate issuer, timestamp, vendor identifier

11. Constraints & Risk Mitigation 

Constraint ID 

Constraint Description 

Mitigation Strategy 

C1: PII Handling 

No raw PII may be stored post-classification. Masking mandatory before persistence. 

Implement Presidio library for PII detection & masking pre-storage. Verify via randomized audits. 

C2: Auditability 

Model must be auditable; black-box-only models unacceptable for regulated data. 

Deploy ensemble combining interpretable models (Logistic Regression, XGBoost) with transformer explanations (SHAP/LIME). 

C3: Deployment 

Support on-premise deployment for air-gapped client environments. 

Package model as containerized service (Docker). Provide offline setup guide & dependency bundle. 

C4: Labeling 

Training data labeling requires dual-approval from Legal & Data Governance. 

Implement labeling workflow with approval gates. Track approver identity & timestamp. Audit disagreements. 

C5: Payments 

Inference API must support mTLS for all payments client integrations (Discover, Visa, Amex). 

Implement mTLS endpoint. Issue client certs per vendor. Validate cert in API gateway. 

 

11.2 Key Risks & Mitigation 

Risk 

Impact 

Mitigation 

Class Imbalance 

Restricted samples rare in real data. Model overfits to majority class. Recall degrades. 

Oversample Restricted class. Use SMOTE synthetic augmentation. Weighted loss function (higher weight for rare class). 

Adversarial Inputs 

Malicious users deliberately obfuscate PII (e.g., 'S.S.N: 123-45-6789'). Model misses. 

Adversarial training with obfuscated examples. Combine model + regex rules. Complement with human-in-loop. 

Model Staleness 

New data types not in training. Model performance drifts over time. 

Active learning pipeline. Monthly retraining. Drift monitoring via Evidently AI. Trigger retraining if F1 drops >3%. 

Misclassification 

Restricted data misclassified as Confidential/Internal. Regulatory liability & breach risk. 

Mandatory human-in-loop for low-confidence predictions. Override mechanism for rule-detected PII. Audit all overrides.
